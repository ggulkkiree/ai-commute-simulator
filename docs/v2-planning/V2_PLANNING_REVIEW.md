# V2 Planning Review

## 검수 범위

검토 대상 문서:

- `V2_REQUIREMENTS.md`
- `SCREEN_FLOW.md`
- `GAME_DATA_DRAFT.md`
- `COMPONENT_PLAN.md`
- `IMAGE_PROMPTS.md`

검수 목적은 v2 설계 문서끼리의 연결성, 충돌 가능성, 핵심 제출 흐름과 보류/제외 가능 기능의 구분을 확인하는 것이다. v1 HTML/CSS/JavaScript 코드는 수정하지 않았다.

## 확인 완료된 항목

### 1. 요구사항과 화면 흐름 연결

- 학생 선택, 학생 확인 모달, 출근 경로 브리핑, AI 출근 계획 입력/결과, 전날 준비, 수면 전환, 아침 알람, 아침 준비, 가방 챙기기, 출근 이동, 결과 컷신, 결과 화면, 교사용 리포트가 `V2_REQUIREMENTS.md`와 `SCREEN_FLOW.md` 양쪽에 반영되어 있다.
- 알람, 10분 더 자기, 아침 준비 활동, 버스 번호 선택, 교통카드 이벤트, 하차벨, 목적지 선택, 결과 판정이 요구사항과 화면 흐름에 모두 존재한다.
- localStorage 기반 학생/히스토리 저장, TTS, AI 힌트, 상단 HUD, 결과 저장, 인쇄 기능은 요구사항에 기록되어 있으며 컴포넌트 계획의 공통 구조로도 대응된다.

### 2. 화면 흐름과 컴포넌트 대응

- `StudentSelect`, `StudentConfirmModal`, `CommuteInfo`, `AIPlanInput`, `AIPlanResult`, `EveningPreparation`, `SleepScene`, `WakeUpScene`, `MorningPrep`, `BagPacking`, `CommuteScreen`, `ResultCutscene`, `ResultScreen`, `TeacherReport`가 주요 화면을 대응한다.
- 통근 화면의 세부 단계는 `BusStop`, `BusRide`, `DestinationMap`, `RouteMap`으로 분해되어 있어 v2 구현 시 단계별 누락을 줄일 수 있다.
- 공통 기능은 `AppShell`, `GameProvider`, `ScreenRouter`, `TopStatusBar`, `ModalRoot`, `TTSButton`, `FeedbackOverlay`로 분리되어 있다.

### 3. 게임 데이터와 화면/컴포넌트 연결

- 화면별 텍스트는 학생 선택, 브리핑, AI 계획, 전날 준비, 수면 전환, 아침 알람, 아침 준비, 출근 이동, 결과, 교사용 리포트 화면에 대응한다.
- 준비물 데이터는 `homeBag`, `EveningBagPacking`, `BagPacking`, `TopStatusBar`, 결과 피드백, 교사용 리포트와 연결된다.
- 버스 번호, 출발 정류장, 노선 정류장, 목적지 데이터는 `CommuteInfo`, `AIPlanInput`, `BusStop`, `BusRide`, `DestinationMap`에 연결된다.
- 결과 메시지와 피드백 태그는 `ResultCutscene`, `ResultScreen`, `TeacherReport`에 연결된다.

### 4. 이미지 목록과 화면 대응

- 학생 선택: 메인 배경, AI 마스코트, 학생 아바타 이미지가 대응된다.
- 브리핑/AI 계획: 경로 지도와 AI 마스코트가 대응된다.
- 전날 준비/수면/아침 알람: 저녁 방, 밤 방, 아침 알람 방 이미지가 대응된다.
- 아침 준비: 방 배경, 샤워/식사/양치/옷 입기 컷신 이미지가 대응된다.
- 가방 챙기기: 열린 가방, 교통카드, 스마트폰, 물병, 우산, 선풍기, 장갑, 겉옷, 마스크, 방해물이 대응된다.
- 통근: 정류장, 버스 내부, 목적지 지도 이미지가 대응된다.
- 결과: 성공, 지각, 실패, 비/배고픔/위생/준비 부족 이미지가 결과 메시지와 대응된다.

### 5. 핵심 로직 연결

- 알람 설정 -> 수면 전환 -> 아침 알람 -> 기상 시간 확정 흐름이 문서 간 일치한다.
- 아침 준비 활동 -> 현재 시간 증가 -> 통근 시작 흐름이 문서 간 일치한다.
- 가방 챙기기 -> 필수/상황/방해물 구분 -> 준비물 피드백 흐름이 문서 간 일치한다.
- 정류장 선택 -> 버스 번호 -> 교통카드 -> 버스 이동 -> 하차벨 -> 목적지 -> 도착 흐름이 문서 간 일치한다.
- 결과 판정은 목표 시간 이하 성공, 목표 + 10분 이내 지각, 10분 초과 실패로 일관된다.

## 문서 간 충돌 가능성이 있는 항목

### 1. 직무 관련 화면의 범위

충돌 가능성:

- `V2_REQUIREMENTS.md`는 "직무 소개와 직무 수행 체크리스트"를 반드시 유지해야 하는 기능에 포함한다.
- `SCREEN_FLOW.md`는 결과 이후 `직무 수행 화면`을 전체 흐름에 포함한다.
- `COMPONENT_PLAN.md`는 `DutyIntro`, `DutyChecklist`를 컴포넌트로 포함한다.
- 그러나 `SCREEN_FLOW.md`의 핵심 제출 흐름 설명은 AI 계획 -> 전날 준비 -> 아침 준비 -> 통근 게임 -> 결과를 주요 흐름으로 설명한다.

검수 의견:

- "오늘의 직무 확인/직무 소개"와 "직무 수행"은 v2 핵심 출근 시뮬레이션 흐름에서 제외하거나 결과 이후 선택 확장 흐름으로 분리하는 것이 안전하다.
- v2 MVP 핵심 흐름에는 넣지 말고, `optional post-result flow`로 표시하는 것을 추천한다.

### 2. 관리자/학생 관리 화면의 범위

충돌 가능성:

- 요구사항에는 학생 관리와 관리자 화면이 유지 기능으로 기록되어 있다.
- `SCREEN_FLOW.md`는 관리자/학생 관리 화면을 전체 화면 흐름 #17로 포함한다.
- 제출용 플레이 흐름에는 관리자 화면이 필수 단계가 아니다.

검수 의견:

- 학생 데이터가 사전에 존재해야 한다면 관리자 화면은 준비/교사용 관리 기능이다.
- v2 핵심 플레이 흐름에서는 제외하고, 별도 교사용 설정 화면 또는 개발/운영 보조 화면으로 분리해야 한다.

### 3. `DutyIntro` 컴포넌트와 화면 흐름 불일치

충돌 가능성:

- `COMPONENT_PLAN.md`에는 `DutyIntro`가 있다.
- `SCREEN_FLOW.md` 전체 흐름에는 "오늘의 직무 확인/직무 소개"가 독립 화면으로 들어가 있지 않고, 결과 후 직무 수행만 있다.

검수 의견:

- `DutyIntro`를 유지하려면 `SCREEN_FLOW.md`에 optional 화면으로 명시해야 한다.
- 핵심 흐름에서 제외한다면 `DutyIntro`는 보류 컴포넌트로 이동하는 편이 좋다.

### 4. 아침 활동 시간 값

충돌 가능성:

- `V2_REQUIREMENTS.md`는 샤워/식사 시간이 v1 데이터 정의와 실행 코드에서 다르게 보인다고 기록했다.
- `GAME_DATA_DRAFT.md`는 활동 텍스트와 선택지만 정리하고, 최종 시간 값을 확정하지 않는다.
- `COMPONENT_PLAN.md`도 활동 컴포넌트에 `minutes` 데이터가 필요하다는 점을 직접 명시하지 않는다.

검수 의견:

- v2 개발 전 샤워 10분/15분, 식사 10분/20분 중 어떤 값을 기준으로 재현할지 결정해야 한다.
- 현재 최종 제출 흐름을 기준으로는 실행 흐름의 10분 값을 우선하는 것이 자연스럽다.

### 5. `bagChecked` 상태의 의미

충돌 가능성:

- `SCREEN_FLOW.md`와 `GAME_DATA_DRAFT.md`는 `morningPrep.bagChecked` 또는 가방 확인 여부를 리포트 데이터로 다룬다.
- `COMPONENT_PLAN.md`는 `BagPacking`에서 `bagItems`, `evePackedItems`, 완료 이벤트를 다루지만 `bagChecked` 확정 규칙을 별도로 정의하지 않는다.

검수 의견:

- v2에서는 "아침 가방 미니게임 완료"를 `bagChecked = true`로 볼지, "필수 준비물을 모두 넣음"을 `bagChecked = true`로 볼지 명확히 해야 한다.
- 추천 기준: 필수 준비물 완료 + 가방 완료 버튼 클릭 시 `bagChecked = true`.

### 6. 10분 더 자기와 `wokeUpEarly`

충돌 가능성:

- `SCREEN_FLOW.md`는 10분 더 자기 후 `wokeUpEarly`를 전달 데이터로 기록한다.
- `V2_REQUIREMENTS.md`는 10분 더 자기와 피드백 우선순위의 늦잠을 기록한다.
- 그러나 `GAME_DATA_DRAFT.md`에는 `wokeUpEarly`가 어떤 조건에서 false가 되는지 명확히 없다.

검수 의견:

- v2에서는 10분 더 자기 선택 시 `wokeUpEarly = false`로 명확히 정의하는 것이 좋다.
- 이 값은 결과 피드백의 `overslept` 후보와 교사용 리포트에 연결되어야 한다.

### 7. `incomplete` 결과 처리

충돌 가능성:

- `V2_REQUIREMENTS.md`는 `incomplete`를 결과 화면에서 성공으로 정규화한다고 기록한다.
- `GAME_DATA_DRAFT.md`의 저장/요약 데이터에는 `judgment`와 통계 항목이 남아 있어 `incomplete`를 별도 통계로 다룰 수 있다.

검수 의견:

- v2에서는 화면 표시용 결과와 저장/분석용 결과를 분리해야 한다.
- 추천 구조: `rawJudgment = incomplete`, `displayJudgment = success`.

### 8. 이미지 목록의 선택/보류 화면 의존성

충돌 가능성:

- `IMAGE_PROMPTS.md`의 `Arrival Workplace`는 회사 도착과 직무 수행 화면 모두에 대응한다.
- 직무 수행을 v2 핵심 흐름에서 제외하면 이 이미지는 도착/결과용으로만 필요하다.

검수 의견:

- 직무 수행을 보류한다면 `Arrival Workplace`의 사용 화면을 "회사 도착/결과 이후 optional 직무"로 조정하는 것이 좋다.

## v2로 가져가야 할 핵심 기능

v2 핵심 제출 흐름에 반드시 포함할 기능:

1. 학생 선택 및 확인
2. 학생 수준별 UI/문장/도움 차이
3. 출근 경로 브리핑
4. AI 출근 계획 입력/확인/계산/결과
5. 전날 알람 설정
6. 수면 전환 컷신
7. 아침 알람과 10분 더 자기
8. 아침 준비 활동: 샤워, 식사, 양치, 옷 입기, 가방 챙기기
9. 필수/상황/방해물 가방 챙기기
10. 통근 게임: 정류장, 버스 번호, 교통카드, 버스 내부, 하차벨, 목적지 찾기
11. 시간 진행과 목표/마감 판정
12. 결과 컷신과 결과 화면
13. 피드백 우선순위와 교사용 빠른 리포트
14. 교사용 상세 리포트와 인쇄
15. TTS, AI 힌트, 상단 HUD
16. 학생/게임 히스토리 저장

## v2에서 보류하거나 제외해도 되는 기능

핵심 제출 흐름에서 제외 또는 보류 가능한 기능:

1. 오늘의 직무 확인/직무 소개 화면
   - 이유: 현재 핵심 출근 시뮬레이션 흐름과 직접 연결되지 않으며, 문서 간 위치가 불명확하다.
   - 권장: optional 또는 post-result 확장으로 분리.

2. 직무 수행 체크리스트
   - 이유: 결과 이후 추가 활동이며 출근 성공/실패 판정에 직접 필요하지 않다.
   - 권장: v2 MVP에서는 보류, 필요 시 결과 이후 선택 흐름으로 유지.

3. 관리자/학생 관리 화면
   - 이유: 제출용 플레이 흐름에는 필수 화면이 아니다.
   - 권장: 교사용 설정/운영 기능으로 분리하고 핵심 플레이 플로우에서는 제외.

4. 레거시 시나리오 화면
   - 대상: `screen-game`, `screen-phase-transition`, `screen-planning`
   - 이유: `SCREEN_FLOW.md`에서도 현재 주요 제출 흐름이 아니라고 표시되어 있다.
   - 권장: v2 핵심 구현 대상에서 제외.

5. 직무별 workplace 준비물 상세 기능
   - 이유: 브리핑/리포트 데이터로는 유용하지만 핵심 통근 게임 판정과 직접 연결되지 않는다.
   - 권장: 데이터는 보존하되 화면 기능은 직무 흐름과 함께 보류.

## 다음 단계 추천

1. v2 핵심 흐름을 한 줄로 확정한다.
   - 추천: 학생 선택 -> 브리핑 -> AI 계획 -> 전날 준비 -> 수면 -> 알람 -> 아침 준비 -> 통근 -> 결과 -> 리포트.

2. `SCREEN_FLOW.md`에서 핵심 흐름과 optional 흐름을 분리한다.
   - 핵심: 출근 시뮬레이션
   - optional: 관리자, 직무 소개/수행, 레거시 시나리오

3. `COMPONENT_PLAN.md`에서 컴포넌트를 `Core`, `Teacher/Settings`, `Optional/Post-result`, `Legacy/Exclude`로 태그한다.

4. 시간 상수를 확정한다.
   - 샤워와 식사 시간은 현재 최종 플레이 흐름 기준 값으로 확정할지 결정해야 한다.
   - 알람 추천 시간, AI 준비 시간, 여유 시간, 버스 시간은 현재 문서 기준으로 일관된다.

5. 상태 필드를 명확히 한다.
   - `rawJudgment` / `displayJudgment`
   - `bagChecked`
   - `wokeUpEarly`
   - `evePrep.bagPacked` / `morningPrep.bagPackedItems`

6. 이미지 목록을 핵심/optional로 다시 태그한다.
   - 직무 관련 이미지는 optional로 표시.
   - 결과/통근/아침 준비 이미지는 core로 표시.

7. v2 개발 전에 데이터 사전을 별도로 확정한다.
   - 화면 텍스트
   - 수준별 문장
   - 준비물 규칙
   - 버스 규칙
   - 결과/피드백 태그
   - 교사용 리포트 필드

## 최종 검수 결론

현재 5개 문서는 v2 재구성을 위한 기본 재료를 충분히 포함하고 있다. 다만 핵심 제출 흐름과 optional/관리/직무/레거시 흐름이 일부 같은 레벨로 섞여 있어, v2 구현 전에 범위 태그를 추가하는 것이 필요하다.

가장 중요한 수정 방향은 다음 세 가지다.

- 직무 소개/직무 수행과 관리자 화면을 v2 핵심 흐름에서 분리한다.
- 아침 활동 시간, 가방 완료 상태, 10분 더 자기 상태, `incomplete` 결과 표시 방식을 명확히 확정한다.
- 컴포넌트와 이미지 목록에 core/optional 태그를 붙여 구현 우선순위를 고정한다.
