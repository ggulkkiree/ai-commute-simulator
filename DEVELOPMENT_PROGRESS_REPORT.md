# 본앤하이리 출근 마스터 개발 진행 보고서

작성일: 2026-06-17  
대상 프로젝트: `bon-highlee-simulator`  
분석 기준: 현재 저장소의 `index.html`, `css/*.css`, `js/*.js` 실제 코드

## 1. 현재 구현된 기능 목록

- 학생 선택 및 학생 이름 검색
  - `js/students.js`에서 localStorage 기반 학생 목록을 관리합니다.
  - `js/app.js`의 `renderMainScreen()`이 학생 카드, 검색, 선택 상태를 렌더링합니다.

- AI 출근 정보 생성
  - `js/state.js`의 `generateTodayCommuteInfo()`가 오늘의 출근 시간, 직무, 날씨, 준비물, 주의사항을 생성합니다.
  - 허용 출근 시간은 `09:00`, `10:00`, `13:00`입니다.
  - 출근 시간에 따라 버스 시간표, 알람 추천, 예상 도착 시간이 계산됩니다.

- 전날 준비
  - `js/eve-prep.js`에서 버스 확인, 가방 챙기기, 잠자기, 알람 설정 흐름을 관리합니다.
  - 알람 선택지는 출근 시간 기준으로 자동 생성됩니다.
  - 가방 선택은 `checkItem()`을 통해 `gameState.bag[item].checked`에 반영됩니다.
  - 다 수준 학생은 2지선다 방식의 준비물 확인 흐름을 사용합니다.

- 아침 알람 및 아침 준비
  - `js/morning-prep.js`에서 알람 울림, 다시 자기, 기상, 샤워/식사/양치/스마트폰/가방 확인, 출발 흐름을 관리합니다.
  - 행동 선택 시 시간이 증가하고, 현재 시간/버스/예상 도착 정보가 갱신됩니다.
  - 행동 완료는 작은 완료 모달로 표시되는 구조가 들어가 있습니다.

- 중앙 시간 계산 엔진
  - `js/state.js`의 `commuteConfig`와 관련 함수가 출근 시간, 실패 기준, 버스 시간표, 이동 시간, 도착 판정을 담당합니다.
  - 이동 기준은 집에서 정류장 10분, 버스 30분, 하차 후 도보 10분입니다.
  - 도착 판정은 출근 시간까지 성공, 출근 시간 이후 10분까지 지각, 그 이후 실패입니다.

- 출근 이동
  - `js/commute.js`에서 집 출발, 정류장 도착, 버스 대기, 버스 탑승, 하차 후 도보, 도착 엔딩 흐름을 처리합니다.
  - 가 수준에서는 조건에 따라 택시 선택이 가능합니다.

- 결과 화면
  - `js/app.js`의 `endGame()`과 `renderResultScreen()`에서 성공/지각/실패 결과, 준비물 상태, 행동 로그, AI 분석 문장을 표시합니다.
  - 결과는 localStorage에 저장됩니다.

- 교사용 보고서
  - `js/report.js`에서 현재 게임 상태 기반 상세 보고서와 학생별 기록 요약을 생성합니다.

- 관리자 화면
  - `js/admin.js`에서 학생 추가, 수정, 삭제, 기록 확인 기능을 관리합니다.

- TTS
  - `js/tts.js`에서 Web Speech API 기반 한국어 음성 안내를 제공합니다.
  - 다 수준 학생에게 자동 음성 안내가 연결되어 있습니다.

## 3. CSS 구조 분석

### 전체 구조

- `css/style.css`는 기본 테마, 화면 공통 스타일, 결과/리포트/관리자, 메인 화면 리디자인, 공통 게임 UI 토큰까지 포함하는 큰 파일입니다.
- `css/wide_vn.css`는 넓은 화면용 레이아웃, 아침 준비, 전날 준비, 출근 이동, 메인 화면 최종 보정 스타일을 추가로 덮어씁니다.
- `index.html`은 `style.css` 다음에 `wide_vn.css`를 로드하므로, 같은 선택자라면 `wide_vn.css`가 우선 적용됩니다.

### 공통 디자인 시스템 상태

`css/style.css`에는 다음 공통 게임 UI 기반이 이미 추가되어 있습니다.

- 색상 토큰: `--game-primary`, `--game-secondary`, `--game-success`, `--game-warning`, `--game-danger`
- 카드: `.game-card`
- 버튼: `.game-button`, `.game-button-primary`, `.game-button-secondary`, `.game-button-success`
- 모달: `.game-modal`, `.game-modal-card`
- 진행 표시: `.game-progress`
- 상단 상태바: `.game-statusbar`
- 배지/드래그 영역: `.game-badge`, `.game-drag-zone`

다만 전체 화면이 아직 이 공통 클래스로 완전히 통일된 것은 아니며, 기존 화면별 전용 CSS와 공존하고 있습니다.

### 중복되는 스타일

- 메인 화면 스타일이 여러 구간에 나뉘어 있습니다.
  - 초기 학생 선택 스타일
  - 메인 화면 리디자인 스타일
  - 강한 게임 타이틀 스타일
  - 디테일 보정 스타일
  - 최종 밀도 조정 스타일

- 전날 준비 스타일도 여러 층으로 존재합니다.
  - 예전 `.eve-prep-container`, `.eve-prep-steps`, `.eve-step-pane`
  - 새 전날 준비 fullscreen 스타일
  - 공통 게임 UI 적용 스타일

- 아침 준비 스타일은 `wide_vn.css` 안에서 기존 스타일과 `Morning Prep v3` 스타일이 겹칩니다.
  - `.morning-prep-container`
  - `.morning-game-stage`
  - `.morning-action-card`
  - `.morning-complete-modal`

- 선택지/버튼 계열 스타일도 화면마다 따로 있습니다.
  - `.btn-start`
  - `.btn-choice`
  - `.game-button`
  - `.alarm-option`
  - `.morning-action-card`

### 사용되지 않거나 사용 빈도가 낮아 보이는 스타일

실제 DOM 기준으로 완전히 삭제해도 된다고 단정하기는 어렵지만, 다음 영역은 현재 주요 흐름에서는 사용 가능성이 낮거나 이전 버전 흔적으로 보입니다.

- 예전 시나리오 선택 화면 관련 스타일
  - `#screen-game`, `.scenario-*`, `.choice-*` 계열 일부

- 예전 전날 준비 구조
  - `.eve-prep-container`, `.eve-step-pane`, `.eve-prep-steps` 일부

- 예전 아침 행동 컷신 구조
  - `.action-cutscene`, `.action-cutscene-img`, `.anim-cutscene-zoom` 계열
  - 현재는 큰 인물 일러스트 대신 작은 완료 모달 방식으로 바뀌어 있습니다.

- 계획 화면 관련 스타일
  - `#screen-planning` 관련 스타일은 HTML/JS에 남아 있으나 기본 시작 흐름에서는 중심 화면이 아닙니다.

### 충돌 가능성이 있는 스타일

- `wide_vn.css`가 나중에 로드되어 `style.css`의 공통 디자인 토큰 적용을 덮어쓸 수 있습니다.
- 일부 화면은 HTML에 inline style이 많아 CSS만으로 수정할 때 우선순위 충돌이 생길 수 있습니다.
- `.btn-start`, `.student-card`, `.feedback-card`, `.bag-dropzone`, `.desk-item`처럼 여러 화면에서 쓰일 수 있는 일반적인 클래스명은 부분 수정 시 다른 화면에 영향을 줄 수 있습니다.
- `!important`가 일부 최종 보정 스타일에 사용되어 있어, 이후 작은 수정도 예상보다 넓게 영향을 줄 수 있습니다.

### 수정하면 위험한 영역

- `wide_vn.css`의 아침 준비 최종 보정 구간
  - 아침 준비 HUD, 행동 카드, 완료 모달, 출발 버튼 위치와 직접 연결되어 있습니다.

- `style.css`의 메인 화면 최종 리디자인 구간
  - 현재 메인 화면의 플래시게임 느낌을 만드는 핵심 영역입니다.

- 결과 화면과 리포트 공통 카드 스타일
  - 학생 결과 화면과 교사용 보고서가 함께 영향을 받을 수 있습니다.

- 전날 준비 가방/알람 관련 스타일
  - 선택 상태, disabled 상태, 진행 표시가 JS 상태와 맞물려 보입니다.

CSS는 전체 재작성보다, 화면별로 “현재 사용 중인 선택자”를 확인한 뒤 짧은 보정 블록을 추가하는 방식이 안전합니다.

## 4. JavaScript 기능 분석

### 현재 작동하는 기능

- 학생 선택 후 게임 시작
- AI 출근 정보 생성
- 출근 시간 기준 알람 선택지 자동 생성
- 전날 가방 준비와 준비물 상태 저장
- 다 수준 2지선다 준비물 확인
- 아침 알람, 다시 자기, 기상 처리
- 아침 행동 선택 시 시간 증가
- 현재 시간 기준 다음 버스, 탈 수 있는 버스, 예상 도착 계산
- 출발 버튼 클릭 후 출근 이동 단계 진입
- 버스 이동과 도착 판정
- 성공/지각/실패 결과 저장
- 결과 화면 표시
- 교사용 보고서 생성
- 관리자 학생 관리
- TTS 안내

### 미구현 또는 부분 구현 상태

- `#screen-duty-intro`는 HTML과 `js/duty.js`에 구현되어 있지만, 현재 `startGame()` 기본 흐름에서는 건너뜁니다.
- `#screen-planning`은 남아 있으나 현재 핵심 흐름에서는 적극 사용되지 않습니다.
- `js/scenarios.js`의 단계별 사회성 시나리오는 이전 흐름으로 남아 있으며, 현재 의도인 “도착 후 바로 엔딩” 구조와는 분리되어 있습니다.
- 결과 화면의 “직무 수행하기”는 성공 후 선택적으로 이어지는 구조입니다. 출근 시간관리 시뮬레이션의 핵심 흐름에서는 부가 기능에 가깝습니다.

### 오류 가능성이 있는 로직

- `gameState.evePrep.busChecked`가 버스 확인 버튼에서 명확히 `true`로 설정되는 코드가 보이지 않습니다.
  - 현재 새 흐름에서는 큰 문제가 아닐 수 있지만, 이전 시나리오 흐름에서 버스 확인 여부를 검사하는 로직과 충돌할 수 있습니다.

- `report.js`의 `generateDetailReport(state)`는 매개변수 `state`를 받지만, 내부 일부 계산은 현재 전역 `gameState` 기반 함수에 의존합니다.
  - 현재 진행 중인 게임 보고서에는 적합하지만, 과거 기록을 상세 보고서로 다시 열 때는 실제 기록과 어긋날 가능성이 있습니다.

- `eve-prep.js`의 알람 선택 시 `gameState.day++`가 사용됩니다.
  - 초기 `gameState` 구조에 `day`가 명시적으로 선언되어 있지 않아 암묵적으로 필드가 생깁니다.

- 오래된 시나리오 흐름과 새 출근 이동 흐름이 함께 남아 있습니다.
  - 현재 사용자는 새 흐름을 타지만, 특정 버튼이나 함수 호출이 예전 흐름으로 연결되면 교육 의도와 다른 화면이 나타날 수 있습니다.

## 5. 가/나/다 수준별 UI 구현 상태

### 다 수준

- 다 수준은 단순 문장, 큰 버튼, TTS 중심으로 구현되어 있습니다.
- 전날 준비물 확인에서 “필요해요 / 필요 없어요” 2지선다 구조가 적용되어 있습니다.
- 아침 준비에서도 필요하지 않은 복잡한 항목을 숨기거나 제한하는 흐름이 있습니다.
- 출발 전 준비물 누락이 있으면 진행을 막는 방식으로 보호가 들어가 있습니다.

보완점:
- 일부 화면에는 여전히 글자 양이 많을 수 있습니다.
- 버스/도착 정보는 계산은 정확하지만, 다 수준 학생에게는 시각적으로 더 단순화할 여지가 있습니다.

### 나 수준

- 현재 가장 균형 잡힌 난이도입니다.
- 준비물 누락이나 시간 부족 상황에서 안내/경고가 제공됩니다.
- 틀렸을 때 바로 실패시키기보다 한 번 더 확인할 수 있는 방향으로 구성되어 있습니다.

보완점:
- 경고 문구와 행동 카드의 우선순위 표시를 조금 더 명확하게 다듬으면 좋습니다.

### 가 수준

- 가 수준은 더 많은 정보와 판단 요소를 볼 수 있습니다.
- 지각 위험 상황에서 택시 선택 같은 비상 대처가 조건부로 제공됩니다.
- 시간, 버스, 출근 목표를 보고 판단하는 구조가 들어가 있습니다.

보완점:
- “연락하기”, “계속 집에 있기” 같은 고난도 판단 선택지는 제한적으로만 구현되어 있습니다.
- 현재는 택시 선택이 대표적인 가 수준 심화 요소입니다.

## 6. TTS 기능 구현 상태

- `js/tts.js`는 Web Speech API를 사용합니다.
- 한국어 음성을 우선 선택하고, 속도는 느리게 설정되어 있습니다.
- `autoSpeak(text, level)`은 다 수준일 때만 자동 재생됩니다.
- 브라우저가 음성 합성을 지원하지 않을 경우 콘솔 경고 후 콜백을 실행합니다.

현재 연결된 곳:

- AI 출근 정보 안내
- 시나리오/피드백 일부
- 전날 준비 안내
- 아침 준비 말풍선의 TTS 버튼

주의점:

- 브라우저 보안 정책상 사용자의 첫 클릭 전에는 자동 음성이 막힐 수 있습니다.
- `morning-prep.js`의 말풍선 TTS 버튼은 `tts.js`의 `speak()`가 아니라 직접 `speechSynthesis`를 사용하는 부분이 있어, 완전한 일관성은 부족합니다.

## 7. 시간 감소 및 선택 결과 로직 구현 상태

내부 로직은 “시간 감소”가 아니라 `advanceTime()`을 통한 현재 시각 증가 방식입니다. 학생 화면에서는 남은 시간, 버스 도착까지 남은 시간, 예상 도착 시간이 줄어드는 형태로 보입니다.

### 중앙 시간 기준

- 출근 목표 시간: AI 출근 정보의 `workStartTime`
- 실패 기준: 출근 목표 시간 + 10분
- 버스 시간표: 출근 시간 기준 60분 전부터 20분 전까지 10분 간격
- 집에서 정류장: 10분
- 버스 이동: 30분
- 하차 후 도보: 10분
- 택시 이동: 20분

### 선택 결과 반영

- 전날 알람 선택: `gameState.evePrep.alarmTime`
- 기상 시각: `gameState.startTime`, `gameState.commute.wakeTime`
- 아침 행동: `gameState.currentTime` 증가, `gameState.actionLog` 기록
- 출발 시각: `gameState.commute.homeDepartureTime`
- 버스 탑승 시각: `gameState.commute.busBoardingTime`
- 도착 시각: `gameState.commute.arrivalTime`
- 결과 판정: `getArrivalStatus()`와 `getJudgment()` 기준

현재 구조는 시간관리 시뮬레이션의 중심축이 비교적 잘 정리되어 있습니다.

## 8. 결과 화면 및 교사용 리포트 구현 상태

### 결과 화면

- 성공, 준비물 미흡, 지각, 출근 실패에 따라 결과 카드가 달라집니다.
- 기상 시각, 집 출발 시각, 버스 탑승 시각, 도착 시각, 준비물 상태가 표시됩니다.
- 놓친 버스, 이동 수단, AI 분석 문장, 행동 로그가 저장됩니다.
- 날씨와 준비물 상태에 따라 결과 이미지가 바뀌는 로직이 있습니다.

### 교사용 리포트

- 학생 정보, 수준, 날짜, 결과 요약을 표시합니다.
- 시간관리, 준비물, 행동 로그, 원인 분석, 지도 포인트를 보여줍니다.
- localStorage의 기록 목록을 기반으로 요약 테이블을 구성합니다.

주의점:

- 현재 상세 보고서는 진행 중인 `gameState`를 기준으로 분석하는 성격이 강합니다.
- 과거 기록을 다시 분석하는 기능을 확장하려면 저장된 기록 객체만으로 보고서를 만드는 구조가 필요합니다.

## 9. 현재 버그 또는 이상 동작

### High

- 전날 버스 확인 상태가 `gameState.evePrep.busChecked`에 명확히 저장되지 않을 가능성이 있습니다.
  - 이전 시나리오 흐름이 다시 사용될 경우 “버스를 확인하지 않았다”는 잘못된 판정이 나올 수 있습니다.

- CSS 중복과 덮어쓰기 위험이 큽니다.
  - `style.css`와 `wide_vn.css`에 같은 화면을 조정하는 스타일이 여러 번 존재합니다.
  - 작은 CSS 수정이 다른 화면에 예기치 않은 영향을 줄 수 있습니다.

- 새 흐름과 이전 흐름이 동시에 남아 있습니다.
  - `screen-game`, `screen-planning`, `scenarios.js`가 여전히 존재하여 유지보수 시 혼동 가능성이 큽니다.

### Medium

- `generateDetailReport(state)`가 전달받은 `state` 대신 전역 상태 계산에 의존하는 부분이 있습니다.
  - 과거 기록 상세 조회 기능을 만들 때 정확도가 떨어질 수 있습니다.

- 아침 준비 완료 모달은 새 방식으로 바뀌었지만, 예전 컷신용 DOM/CSS/이미지 흔적이 남아 있습니다.
  - 기능 오류는 아니지만 구조를 읽기 어렵게 만듭니다.

- HTML inline style이 많아 공통 디자인 시스템 적용이 어렵습니다.
  - 특히 전날 준비와 아침 준비 화면은 CSS만으로 통일하기가 까다롭습니다.

- `gameState.day++`처럼 초기 상태에 없는 필드가 사용됩니다.
  - 현재 큰 오류는 아니지만 상태 구조가 느슨해집니다.

### Low

- PowerShell 환경에서 한글 출력이 깨져 보일 수 있습니다.
  - 실제 브라우저 표시 문제와 개발 도구 출력 인코딩 문제를 구분해야 합니다.

- TTS 호출 방식이 일부 화면에서 `tts.js`와 직접 `speechSynthesis` 호출로 나뉘어 있습니다.
  - 기능은 동작하지만 관리 포인트가 분산되어 있습니다.

- 사용 빈도가 낮은 CSS와 이미지 자산이 남아 있어 프로젝트가 무거워 보입니다.
  - 바로 삭제하기보다는 실제 사용 여부 확인 후 단계적으로 정리하는 것이 안전합니다.

## 10. 앞으로 수정해야 할 우선순위

### High

1. 전날 버스 확인 상태 저장 보강
   - `eve-prep.js`에서 버스 확인 완료 시 `gameState.evePrep.busChecked = true`가 확실히 저장되도록 확인합니다.

2. 현재 기본 흐름과 이전 흐름의 경계 정리
   - `screen-game`, `screen-planning`, `scenarios.js`가 현재 수업 흐름에서 필요한지 표시하거나, 임시 비활성 상태임을 코드 주석과 문서로 명확히 해야 합니다.

3. CSS 충돌 방지 기준 만들기
   - 전체 CSS를 갈아엎지 말고, 화면별 “수정 가능 구간”과 “건드리면 위험한 구간”을 주석으로 구분하는 것이 좋습니다.

### Medium

1. 리포트 생성 함수의 입력 기준 정리
   - 현재 게임 상태용 보고서와 저장된 기록용 보고서를 분리하면 안전합니다.

2. 아침 준비 화면의 이전 컷신 잔여 코드 정리
   - 사용하지 않는 큰 인물 일러스트 흐름과 관련 CSS를 바로 삭제하지 말고, 실제 DOM 사용 여부를 확인한 뒤 정리합니다.

3. 공통 디자인 시스템을 화면별로 천천히 적용
   - 전날 준비, 아침 준비, 결과 화면 순서로 작은 단위 적용이 좋습니다.

4. TTS 호출 방식 통일
   - `tts.js`의 `speak()`를 공통 진입점으로 쓰면 유지보수가 쉬워집니다.

### Low

1. 사용하지 않는 이미지와 CSS 후보 목록화
   - 삭제 전 목록만 만들고, 실제 삭제는 별도 작업으로 분리하는 것이 안전합니다.

2. 관리자 화면 스타일 소폭 정리
   - 현재 기능 유지가 우선이므로 UI 통일은 후순위가 좋습니다.

3. 코드 주석과 프로젝트 가이드 갱신
   - 새 시간 계산 엔진과 화면 흐름을 문서에 반영하면 다음 유지보수가 쉬워집니다.
